理解 JS 中的 generators

这篇文章深入探讨了 JavaScript 中的生成器（generators），强调其作为”pull streams”的特性，并详细阐述了其适用场景。作者首先区分了”push streams”和”pull streams”，指出前者的数据传递由外部控制，如 WebSocket、从磁盘读取文件和服务器发送事件等；而后者的数据获取由内部控制，即消费者决定何时获取数据。生成器正是实现”pull streams”的工具，它允许函数的执行被暂停和恢复，从而在需要时生成数据。文章还讨论了惰性（lazy）和急切（eager）求值的概念，强调生成器采用惰性求值，仅在需要时才计算值。在适用场景方面，作者列举了生成器的三个主要用途：惰性求值（按需生成或处理大型或无限的数据集）、异步编程（处理异步操作）和迭代器（允许在复杂流程中暂停和恢复执行）。通过这些特性，生成器为 JavaScript 开发者提供了强大的工具，以更灵活地控制数据流和流程管理。

如何管理 NodeJS 中多线程内存

在 Node.js 多线程环境中，数据共享的主要问题 在于竞态条件（Race Condition）和数据一致性。由于多个 Worker 线程可以通过 SharedArrayBuffer 访问共享内存，
并发读写 可能导致数据不一致，尤其是在多个线程同时修改某个值时，可能出现覆盖、丢失更新等问题。为了解决这些问题，文章提出了 两种解决方案：
第一，使用 Atomics API（如 Atomics.add、Atomics.compareExchange），确保对共享内存的操作是 原子的，避免多个线程同时修改数据时引发的不确定行为；
第二，构建 同步原语（Semaphore 和 Mutex），通过 等待、锁机制 控制多个线程对共享资源的访问，类似于 Golang 的 WaitGroup、Mutex，确保线程按照安全的顺序访问数据，从而保证数据一致性和线程安全。

不要用 TypeScript 的枚举

NodeJS 23 发布了，新版本与 Bun 和 Deno 一样，支持直接运行 TypeScript，不再需要二次编译。
但是对 TS 的特有语法存在限制，例如枚举与namespace。因为它们会被转译为 JavaScript 对象。
因此，TypeScript 团队通过新的编译器标志 –erasableSyntaxOnly，通过禁用这些属性，帮助用户更好的在 Node 环境下运行 TS。
